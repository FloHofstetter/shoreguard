# M12 — In-k8s Federation Runbook

The k8s analog of [`m8-demo.md`](./m8-demo.md): the same two-gateway
federation story, but every component (ShoreGuard + both OpenShell
gateways) runs as a k8s pod, traffic flows over Service DNS with mTLS,
and registration reads the client cert from a chart-exported Secret.

> **Note — this is an internal regression runbook, not a production
> deployment guide.** The `tests/fixtures/charts/openshell-cluster`
> chart used here is an internal test fixture that wraps NVIDIA's
> `ghcr.io/nvidia/openshell/cluster` all-in-one image. For actual
> production installs of OpenShell alongside ShoreGuard, install
> NVIDIA's upstream OpenShell Helm chart separately and register the
> gateway endpoints via the ShoreGuard UI or API. ShoreGuard does not
> own or ship NVIDIA's OpenShell chart.

## What this proves

- `charts/shoreguard` and `tests/fixtures/charts/openshell-cluster`
  compose into a working federation story in a single kind cluster
  without any host processes besides `kubectl port-forward`.
- ShoreGuard's gateway client reaches OpenShell via
  `<release>-openshell-cluster.<ns>.svc.cluster.local:30051` with mTLS
  material generated by the chart's post-install bootstrap Job.
- The federation surfaces (`?label=env:…`, per-gateway audit
  attribution, unfiltered audit coalescence, `/api/gateway/list`) work
  identically to the host-process M8 flow.
- Rolling one gateway's StatefulSet does not disrupt traffic on the
  other (Phase K).

If any phase fails, that is the M12 gap to fix.

## Prerequisites

- A kind cluster (see the M11 runbook for the canonical config).
- Helm 3.12+, kubectl 1.28+.
- `uv sync` clean in the ShoreGuard repo (local Python 3.12+ for
  `scripts/m12_demo.py`).
- Real Anthropic API key in `ANTHROPIC_API_KEY`.
- `charts/shoreguard` chart available at `./charts/shoreguard`
  (v0.30.1+ appVersion).
- Internal fixture chart available at
  `./tests/fixtures/charts/openshell-cluster`.

## Phase 0 — Bring up the stack

```bash
kind create cluster --name sg-m12

kubectl create ns shoreguard
# If your cluster enforces Pod Security admission, the openshell-cluster
# pod needs privileged. Label the namespace explicitly so install does
# not fall over later:
kubectl label ns shoreguard \
  pod-security.kubernetes.io/enforce=privileged \
  pod-security.kubernetes.io/warn=privileged \
  pod-security.kubernetes.io/audit=privileged

kubectl -n shoreguard create secret generic sg-secrets \
  --from-literal=admin-password=m12-demo-pass \
  --from-literal=secret-key=$(openssl rand -hex 32)

# Two in-cluster OpenShell gateways. Each helm install triggers a
# post-install bootstrap Job that generates mTLS material inside the
# cluster pod and exports the client cert to <release>-openshell-cluster-client-tls.
helm install cluster-dev ./tests/fixtures/charts/openshell-cluster \
  --namespace shoreguard --set label.env=dev
helm install cluster-staging ./tests/fixtures/charts/openshell-cluster \
  --namespace shoreguard --set label.env=staging

kubectl -n shoreguard wait --for=condition=Complete job \
  -l openshell.io/bootstrap=true --timeout=300s

# Sanity: both client-cert Secrets must exist.
kubectl -n shoreguard get secret \
  cluster-dev-openshell-cluster-client-tls \
  cluster-staging-openshell-cluster-client-tls

# ShoreGuard (production values, single replica, ingress disabled —
# Phase 4 uses port-forward).
helm install sg ./charts/shoreguard \
  --namespace shoreguard \
  --set existingSecret=sg-secrets \
  --set persistence.enabled=true
kubectl -n shoreguard wait --for=condition=ready pod \
  -l app.kubernetes.io/name=shoreguard --timeout=180s

helm test cluster-dev   -n shoreguard
helm test cluster-staging -n shoreguard
helm test sg            -n shoreguard
```

## Phase 0.5 — Port-forward to ShoreGuard

The demo script talks HTTP to ShoreGuard. Open a dedicated terminal:

```bash
kubectl -n shoreguard port-forward svc/sg-shoreguard 8888:8888
```

Health check:

```bash
curl -sf http://127.0.0.1:8888/healthz
curl -sf http://127.0.0.1:8888/version
```

## Phase K — Rolling-restart one gateway, watch the other

The M12 acceptance criterion: a `rollout restart` on one StatefulSet
must not disrupt traffic on the other. In two terminals:

```bash
# Terminal A — drive staging traffic in a loop
while true; do
  curl -sf -b /tmp/sg.cookies http://127.0.0.1:8888/api/audit\?gateway=cluster-staging\&limit=1 \
    -o /dev/null && echo "$(date +%H:%M:%S) staging OK" || echo "$(date +%H:%M:%S) STAGING BROKE"
  sleep 2
done

# Terminal B — bounce cluster-dev
kubectl -n shoreguard rollout restart statefulset/cluster-dev-openshell-cluster
kubectl -n shoreguard rollout status statefulset/cluster-dev-openshell-cluster --timeout=240s
```

Expected: the loop in Terminal A never prints `STAGING BROKE` while the
rollout is in-flight. The ShoreGuard API stays available throughout
because only the federated `cluster-dev` gateway is affected, and
`cluster-staging`'s gRPC channel is independent.

## Cleanup

```bash
helm -n shoreguard uninstall sg cluster-dev cluster-staging
kubectl delete ns shoreguard
kind delete cluster --name sg-m12
```

## Notes

- **`openshell` CLI is not required on the host.** Every sandbox-exec
  step in `scripts/m12_demo.py` goes through ShoreGuard's
  `POST /api/gateways/{gw}/sandboxes/{sb}/exec` LRO endpoint, so the
  test host only needs `kubectl`, `helm`, and `uv`.
- **`SHOREGUARD_LOCAL_MODE`** is **off** in the chart default; the
  local-mode auto-register-mtls bug described in
  [`docs/roadmap`](../docs) cannot fire here because the chart never
  sets that flag.
- **mTLS material lifetime** is 10 years (self-signed). This chart is
  not a production cert-rotation story; M12 is about proving the
  topology works.
- **Cross-namespace federation** (ShoreGuard in `ns=a`, gateways in
  `ns=b`) is out of scope. All three releases must share a namespace.
